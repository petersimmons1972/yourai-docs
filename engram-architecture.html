Container Topology

Three Docker containers collaborate in a private Compose network. The engram-go-app process is the only externally reachable component, exposing a single HTTP endpoint on port 8788. PostgreSQL and Ollama are entirely internal — never accessible from the host network — and are consumed only by the application layer.

Protocol Flows

Every AI-assisted memory operation follows one of these paths. The MCP transport is HTTP with Server-Sent Events for streaming — no WebSocket, no custom protocol. The client connects once and the server pushes tool results as they complete.

Claude Code → HTTP+SSE · :8788/mcp → engram-go-app → HTTP REST · :11434 → Ollama memory store / recall (embed query)
Claude Code → HTTP+SSE · :8788/mcp → engram-go-app → PostgreSQL wire · :5432 → Postgres ANN vector search + full-text
engram-go-app → HTTPS (opt-in) → Claude API memory_reason / memory_summarize (when ENGRAM_CLAUDE_SUMMARIZE=true)
Ollama → ROCm / HIP → AMD GPU nomic-embed-text inference + llama3.2:3b summarization

Container Details

All three containers start in order: postgres must pass a pg_isready health check, then ollama must confirm nomic-embed-text is loaded, then engram-go-app starts and passes its own /health probe. Any failure in the chain blocks startup cleanly.

Database Schema

The engram database has grown through 18 migrations from a simple key-value store to a full episodic, graph-connected, vector-indexed memory system. Every memory carries a UUID v7 primary key, full-text search via tsvector, and a 768-dimensional pgvector embedding stored in an HNSW index for sub-millisecond approximate nearest-neighbour retrieval.

CORE memories
Primary store — content, type, project, tags, importance, access counts, expiry, immutability, episode linkage. Full-text search via generated tsvector column.
VECTOR chunks
Sentence-window chunks of each memory. Stores 768-dim embedding_vec vector(768) with HNSW index. Used for ANN semantic retrieval. Cascades on memory delete.
GRAPH relationships
Directed weighted edges between memories. Relation types: caused_by, depends_on, supersedes, contradicts, supports, derived_from, part_of, and more.
META project_meta
Per-project key/value configuration store. Composite PK (project, key) prevents cross-project collision. Stores schema version, backfill state, and feature flags.
EPISODE episodes
Session grouping — memories from a single AI session are bound to an episode ID for chronological recall. Episodes carry start/end timestamps and optional summaries.
DOCS documents
Tier-2 raw document storage for large inputs (>8 MB). Parent memory holds a synopsis + foreign key. SHA-256 deduplication. Up to 50 MB per document.
AUDIT retrieval_events
Logs every recall operation with query, scores, and feedback classification. Powers adaptive weight tuning and retrieval drift monitoring over time.
ENTITY entities
Canonical entity index extracted from memory content. Supports entity-centric queries and relationship inference across the memory graph.
WEIGHT weight_history
Time-series of adaptive retrieval weight updates per project. Used to replay weight evolution and diagnose ranking drift via memory_audit_compare.

How It All Fits Together

Engram's design philosophy is locality first. All three containers run on the same Docker host, communicating over a private bridge network with no external exposure beyond port 8788. The Ollama container is bound to 127.0.0.1 — it is invisible to the wider network even if the host firewall is misconfigured.

When an AI client like Claude Code wants to store a memory, it calls memory_store over the MCP connection. The Go application receives the request, embeds the content by calling Ollama's REST API (/api/embed), chunks the text into sentence windows, and writes both the memory record and the chunk embeddings to PostgreSQL in a single transaction. The pgvector HNSW index makes future memory_recall calls — which combine ANN vector search with full-text ts_rank scoring — complete in under 5 ms even with hundreds of thousands of stored memories.

The episodic layer (migration 008) binds every SSE session to a named episode so that memories from a single working session can be retrieved as a chronological unit — useful for session summaries and context recovery after compaction. The relationship graph (relationships table) lets the application infer semantic connections between memories using memory_sleep, which runs a background consolidation pass and creates typed directed edges between semantically related entries.

The optional Claude API integration adds four additional tools that synthesise stored memories into grounded natural-language answers. These are off by default and activated per-project via environment flags, so the system remains fully functional on a fully air-gapped host with only the Ollama models available.
